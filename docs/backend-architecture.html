二、代码分层架构

项目采用清晰的五层分层，上层依赖下层，职责边界明确。路由层只做鉴权 / 限流 / 响应包装，业务逻辑下沉到服务层，RAG 流水线独立编排。

图 1 · 后端代码五层分层架构

各层职责

1. API 路由层 router/

定义 5 个 APIRouter：chat（Agent SSE 流式 / RAG 检索 / 会话管理）、knowledge（文档上传 / 删除 / 切片查看 / 图片服务）、user、feedback、health。统一使用 Depends(get_current_user_id) 鉴权 + rate_limit 限流，返回统一 success_response 包装。

2. 业务服务层 services/ + router/*_service.py

DatabaseSessionManager 负责会话创建 / 消息持久化 / 思考事件存储；ChatService / KnowledgeService 作为路由与底层服务的中介。

3. RAG + Agent 核心层 rag/ + agent/

五阶段检索流水线（见下节）+ LangGraph StateGraph Agent 编排 LLM 节点与工具节点。

4. 数据层 models/ + schemas/ + db/

ORM 定义 ChatSession / ChatMessage / ChatThinkingEvent / QueryLog；Pydantic schema 定义请求 / 响应契约；MySQL async engine + Redis async client。

5. 基础设施层 core/ + utils/ + cache/

日志、限流、异常处理、JWT 鉴权、配置加载、工厂模式创建模型实例。

三、RAG 五阶段检索流水线

RAG 核心由 rag_service.py 编排，采用五阶段流水线——不是简单的"向量检索 + LLM 生成"，而是工程化的多路检索 + 精排 + 重排 + 总结。

图 2 · RAG 五阶段检索流水线

各阶段详解

  1. 查询预处理：QueryProcessor 生成子查询变体 + HyDE 假设文档，扩展查询语义。
  2. 多路检索（粗排）：Milvus 向量（1024 维）+ BM25（jieba 分词）并行检索，通过 RRF 融合；动态权重——精确查询偏向 BM25，语义查询偏向向量。
  3. 精排：ReorderService 用 CrossEncoder（bge-reranker-large）计算相关性分数。
  4. 多因素排序：MultiFactorRanker 结合相关性、时间衰减、文档权重综合排序。
  5. 扩展总结：相邻 chunk 扩展 + PromptTemplate | ChatModel | StrOutputParser 分批摘要。

四、核心模块说明

向量数据库（Milvus）

Milvus 维护两个 collection：

  • rag_collection — 文本向量，1024 维，IVF_FLAT + COSINE，user_id 作为分区键实现用户隔离。
  • rag_image_collection — CLIP 视觉向量，512 维，支持跨模态图片检索（文本 → CLIP 编码 → 图片向量搜索）。

LangChain / LangGraph 集成

utils/factory.py 自定义 ChatModel（继承 BaseChatModel），统一封装 DeepSeek / 阿里云 / Ollama 为 OpenAI 兼容接口，支持工具绑定与流式输出。Agent 用 LangGraph StateGraph 构建 llm → tools 条件循环图，共 6 个工具：

RAG 摘要 SQL 查询 天气 时间 用户信息 重排序

检索器策略模式

rag/retrievers/ 采用策略模式，可灵活替换：

  • milvus_retriever — 向量检索
  • bm25_retriever — 关键词检索（中文 jieba 分词）
  • rrf_retriever — RRF 融合检索
  • empty_retriever — 空检索（降级用）

鉴权机制

JWT 由独立的 DjangoUserService（:8001）签发，本服务仅验签（HS256）。Redis 存储 JWT 黑名单（jti），用户信息缓存于 Redis（1 小时过期），缺失时降级调用 Django API。

数据持久化

  • MySQL — 会话 / 消息 / 思考事件 / 查询日志
  • Milvus — 向量数据
  • Redis — 缓存 + JWT 黑名单 + 限流计数
  • 文件系统 — PDF 提取图片（data/extracted_images/{user_id}/{md5}/）

全链路 async：SQLAlchemy async + aiomysql + Redis asyncio，配合令牌桶限流，适合高并发。

五、Docker 部署架构

docker-compose.yml 编排服务，组成 rag-net 桥接网络；后端、前端、DjangoUserService 为三个独立进程分别部署。

图 3 · Docker 部署拓扑

部署要点

  • 容器编排：docker-compose.yml 在 rag-net 桥接网络内编排 MySQL 8.0、Redis 7、Milvus standalone、FastAPI backend。
  • Milvus 全家桶：standalone 模式依赖 etcd（元数据）+ minio（对象存储）。
  • 后端镜像：基于 python:3.12-slim，用 uv 安装依赖，暴露 8000 端口。
  • 外部独立服务：前端 Vue3（:3000）、DjangoUserService（:8001）、LLM 提供商。
  • 数据持久化：本地 volumes/ 目录持久化 Milvus 相关数据（etcd / milvus / minio）。

六、测试结构

tests/ 为扁平结构，共 12 个测试文件 + conftest.py（pytest fixtures）。

  • 配置：pyproject.toml 设置 asyncio_mode=auto。
  • 三类 marker：e2e（端到端）、db（需数据库）、slow（模型加载）。
  • 覆盖范围：BM25 中文分词、文档处理器（code / excel / pdf headings）、检索流水线、反馈、语言检测、文本切片等模块。
