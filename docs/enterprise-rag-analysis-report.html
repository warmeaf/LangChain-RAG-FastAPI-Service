2.1 文档切分：不同文件类型的处理策略

📋 企业级标准要求

  • Excel：行→自然语言描述，多 Sheet 处理，合并单元格/多级表头解析
  • PDF：纯文本/扫描版/混合版三类分治，OCR + 后处理纠错
  • Word/PPT：格式保留（Markdown），PowerPoint 按幻灯片逻辑切分
  • 代码：AST 解析，按函数/类/方法切分，保留上下文
  • 邮件/聊天：保留对话线程和元信息
  • 多种切分策略组合：段落/标题层级/语义单元/完整观点/滑动窗口

✅ 当前实现

  • Excel: excel_processor.py — 实现了行→自然语言（"表头值"），支持多 Sheet（sheet 名作上下文），但未处理合并单元格和多级表头
  • PDF: ocr_processor.py — PaddleOCR 扫描版识别 + 后处理纠错（常见错字字典）；混合版通过 pdf_multimodal_loader 支持
  • Word/PPT: format_preserver.py — 使用 Markdown 语法保留格式（Title→#, Header→##, ListItem→-），PPT 通过 unstructured partition 解析
  • 代码: code_processor.py — Python 用 AST 按 FunctionDef/ClassDef 切分并保留类上下文；其他语言退化为空行分块（弱）
  • 邮件/聊天: ❌ 未实现
  • 切分策略: 递归字符切分（中文感知分隔符）+ 语义合并优化（embedding 相似度 >0.7 合并）

3.1 长问题的处理：压缩和改写

📋 企业级标准

  • 超长问题 → LLM 压缩保留核心信息
  • 复杂问题 → 拆分为多个子问题
  • 两者结合使用

✅ 当前实现

  • 压缩: query_processor.py:61-68 — 长度 > max_length(400) 时触发 LLM 压缩
  • 拆解: query_processor.py:70-80 — LLM 将复合问题拆分为最多 3 个子问题
  • 扩展: query_processor.py:82-92 — 每个子问题生成最多 2 个语义变体
  • 三段式流水线: 压缩 → 拆解 → 扩展 → 并行检索

💡 超出标准：实现了"查询扩展"（第三章 3.6 的内容）作为查询预处理的第三阶段，形成了"压缩→拆解→扩展→并行检索+合并去重"的完整查询增强流水线。还将 HyDE（假设文档嵌入）作为额外检索变体（rag_service.py:69-70），进一步提升了召回覆盖面。

3.2 召回率优化：粗排和精排（两阶段检索）

📋 企业级标准

  • 粗排: 向量检索召回 Top-100
  • 精排: Rerank 模型对 Top-100 重打分取 Top-10
  • 粗排目标"不漏"，精排目标"准确"

✅ 当前实现

  • 粗排: coarse_k=100（rag.yaml:4），混合检索（Milvus向量 + BM25关键词 + RRF融合）
  • 精排: reorder_service.py — BAAI/bge-reranker-large 对粗排结果逐个打分
  • 完整两阶段流程: 粗排(100) → Reranker精排 → 多因素排序 → Top-N(8) 总结

  • 支持 GPU/CPU 自适应（reorder_service.py:13）

3.4 综合排序：多因素加权

📋 企业级标准

  • 最终得分 = 相似度×0.5 + 时间衰减×0.3 + 权重×0.2
  • 时间衰减: 指数衰减 (今天的文档=1.0, 1年前=0.37)
  • 权重: 人工标注的文档重要性

✅ 当前实现

  • 完全对齐: w_relevance=0.5, w_time=0.3, w_weight=0.2（rag.yaml:12-14）
  • 指数衰减: exp(-λ * years_ago), λ=1.0（multi_factor_ranker.py:42）
  • 文档权重: 从 metadata.doc_weight 读取，默认 1.0

3.5 混合检索：向量检索 + 关键词检索

📋 企业级标准

  • 向量检索 + 关键词检索（Elasticsearch）
  • 合并去重后精排
  • 根据问题类型动态调整权重

✅ 当前实现

  • Milvus 向量检索（milvus_retriever.py）+ BM25 关键词检索（hybrid_retriever.py:13-34）
  • RRF 融合（Reciprocal Rank Fusion, k=60）：无需手动调权，自动平衡两路结果
  • BM25 使用 rank-bm25 库，在内存中构建索引
  • 动态权重接口预留但返回固定值 (0.5, 0.5)

📝 总体评价

🎯 核心结论：该项目已经实现了企业级 RAG 标准的 核心骨架。在两阶段检索（粗排+精排）、多因素排序、查询预处理、向量数据库选型等关键领域达到了高度对齐甚至超越标准（如 HyDE 集成）。代码架构清晰、配置化管理完善、多租户设计和异步处理具备生产级基础。

🌟 三大突出优势：
  1. 检索流水线完整性：查询预处理(压缩+拆解+扩展+HyDE) → 混合粗排(向量+BM25+RRF) → Reranker精排 → 多因素排序，形成了教科书级的企业 RAG pipeline。
  2. 模型选型精准：bge-large-zh(Embedding) + bge-reranker-large(Reranker) 是中文场景的最佳实践组合。
  3. 工程化程度高：配置化参数、启动预热、SSE 流式进度、MD5 去重、多租户隔离等工程实践扎实。

⚠️ 三个关键短板：
  1. 负反馈闭环断裂：反馈数据收集完备但未驱动检索优化，是企业级标准中评分最低的领域（38%）。
  2. 代码文件处理不均衡：Python 有 AST 支持但 JS/TS/Java/Go 退化为空行分块，生产环境中代码库检索体验会受影响。
  3. 混合检索动态权重缺失：向量+BM25 融合使用固定 RRF 策略，未根据问题类型动态调整，对精确匹配类查询可能召回不足。
