# 升级计划

> 本文档由 39 轮架构拷问（grilling）驱动生成，每一项决策均有明确依据。目标读者为执行升级的 AI Agent，要求需求明确、方向无误、可执行落地、可自行验证。

---

## 一、升级目标

将当前 RAG 检索系统全面升级为基于 AI Agent 的内容检索系统。核心是 **Plan-then-Execute Agent Loop**：Agent 收到用户提问后，先制定检索计划，再逐步执行，执行过程中审视每一步的结果，必要时修正计划，最后汇总回答。

---

## 二、技术架构决策

### 2.1 Agent 框架

- **选型**：LangGraph StateGraph + `langchain_anthropic.ChatAnthropic`
- **协议**：Anthropic 原生 Messages API（`tool_use` / `tool_result` content blocks）
- **模型端点**：`https://api.deepseek.com/anthropic`（DeepSeek 官方 Anthropic 兼容端点）
- **认证**：`x-api-key` header（Anthropic 标准）
- **模型名**：`deepseek-v4-flash`
- **上下文窗口**：1,000,000 token
- **兼容性说明**（来自 DeepSeek 官方文档）：
  - `tool_use` / `tool_result` content blocks：完全支持
  - `disable_parallel_tool_use`：被忽略——模型可能一次返回多个 `tool_use`，Execution 节点逐个执行
  - `is_error` on `tool_result`：被忽略——工具报错通过 content 文本传递
  - `thinking`：支持（`budget_tokens` 被忽略）
  - `image` / `document` content type：不支持
- **说明**：当前 RAG 流水线（摘要生成）继续使用 OpenAI 兼容协议（`factory.py` 的 `ChatModel`），与新 Agent 的 Anthropic 协议共存，互不污染

### 2.2 Agent 循环模式

**Plan-then-Execute**，LangGraph 图结构：

```
START → Planning → Execution → Summarization → END
           ↑            │
           └── replan ──┘ (最多 3 次)
```

- **Planning 节点**：LLM 调用 `create_plan` 工具输出结构化检索计划
- **Execution 节点**：串行逐步执行计划，每步完成后 LLM 调用 `evaluate_step_result` 工具审视结果，决策 continue/replan/skip
- **Summarization 节点**：所有步骤完成后，独立 LLM 调用生成最终回答（不走工具，直接文本输出）

### 2.3 执行约束

- **严格串行**：每次只有一个正在执行的步骤，不并行
- **多个 tool_use 处理**：DeepSeek 的 `disable_parallel_tool_use` 被忽略，模型可能一次返回多个 `tool_use`。Execution 节点遍历 `tool_use` blocks，逐个执行工具，全部完成后进入审视
- **每步审视**：每个 Step 执行完后，工具结果交给 LLM 审视，自行决策是否继续/修正/跳过
- **修正上限**：最多 replan 3 次，超过则强制进入 Summarization
- **允许跳过工具**：若 LLM 判断问题不需要检索，可直接进入 Summarization
- **全局超时**：120 秒，超时中断并返回已收集结果

## 三、Agent State 数据模型

```python
class Step(TypedDict):
    id: str              # 步骤唯一标识
    tool_name: str       # 工具名
    tool_args: dict      # 工具参数
    reason: str          # 为什么需要这个步骤
    depends_on: list[str]  # 依赖的 step id 列表
    status: str          # pending | running | done | failed | skipped
    result: str | None   # 工具返回结果（执行后填充）

class AgentState(TypedDict):
    messages: list       # 对话历史（Anthropic content blocks）
    plan: list[Step]     # 当前检索计划
    current_step: int    # 当前执行到第几步
    replan_count: int    # 已重规划次数
    tool_results: dict   # 工具执行结果汇总 {step_id: result}
    final_answer: str    # 最终回答（Summarization 节点产出）
```

- **messages 管理**：手动 append，不使用 LangGraph 的 `MessagesState` 自动合并
- **user_id**：系统内部从 ContextVar 获取并注入工具参数，LLM 不可见不可改

## 四、工具清单（8 个）

### 4.1 Vector Search（向量检索）
- **来源**：从 `rag_summary_tools` 拆分，只做检索不含摘要
- **输入**：`query: str`，`user_id: str`（系统注入）
- **输出**：检索到的 top-K chunk 拼接文本，每个 chunk 带 `[来源: xxx, 相关度: 0.xx]` 前缀
- **内部能力**：查询预处理、HyDE 生成、dense 向量检索、重排序（bge-reranker）、chunk 扩展——均内嵌在工具内部，不暴露给 Agent

### 4.2 Keyword Search（关键词检索）
- **来源**：从 `MilvusHybridRetriever` 拆出 BM25 分支，封装为独立工具
- **输入**：`query: str`，`user_id: str`（系统注入）
- **输出**：BM25 检索到的 top-K chunk 拼接文本（格式同 Vector Search）
- **内部能力**：基于 Milvus 2.5 内置 BM25 Function（`sparse` 字段 + `SPARSE_INVERTED_INDEX`）

### 4.3 SQL Query（只读 SQL 查询）
- **来源**：重写 `execute_readonly_sql`，同时合并原计划的 Metadata Filter (MySQL)
- **输入**：自然语言查询意图描述（如"技术文档类别下权重高于 0.8 的文档有哪些"），`user_id: str`（系统注入）
- **输出**：查询结果的结构化文本
- **安全**：工具内部将自然语言转 SQL + 强制注入 `user_id` 过滤 + 仅允许 SELECT，拒绝 DDL/DML
### 4.4 Metadata Filter（Milvus 标量过滤）
- **来源**：全新
- **输入**：自然语言过滤条件描述（如"doc_entity 是张三且创建于 2024 年之后"），`user_id: str`（系统注入）
- **输出**：过滤结果的文档计数和统计摘要 + 匹配文档 id 列表
- **内部**：工具内部将自然语言转 Milvus expr 表达式（`doc_entity == "..." && created_at > ...`）
- **可用标量字段**：`id`, `text`, `user_id`, `doc_entity`, `doc_weight`, `created_at`, `metadata`(JSON)

### 4.5 Weather API（天气查询）
- **来源**：重写 `get_weather_tools`，接真实 API
- **提供商**：和风天气（QWeather）
- **环境变量**：`QWEATHER_API_KEY`（从 `.env` 加载，系统注入工具）
- **输入**：`city: str`
- **输出**：该城市天气信息的文本描述

### 4.6 Time（时间获取）
- **来源**：重写 `what_time_is_now`
- **输入**：无
- **输出**：当前年月日时分的文本描述

### 4.7 OCR（光学字符识别）
- **来源**：包装现有 PaddleOCR 能力
- **输入**：`file_path: str`（文件路径）
- **输出**：识别出的文本内容
- **安全**：工具层校验路径合法性，防止路径遍历攻击
- **说明**：也可在内部文档解析时使用（非 Agent 调用路径）

### 4.8 已移除的工具
- **Analytics Tool**：不再需要
- **Code Interpreter**：不再需要

### 4.9 工具定义规范

- 系统提示词中：用 Anthropic 风格 XML 描述工具（`<tool name="...">` 含 `<description>` 和 `<parameters>`）
- API 请求中：`tools` 参数传 JSON Schema
- 两份定义从同一源生成（代码中的函数签名 + docstring），保持一致性
- LLM 自主决定工具的调用顺序和组合，不预设固定流程

## 五、元工具（Agent Loop 内部控制）

这三个工具不在工具清单中暴露给用户视角，而是控制 Agent 自身行为的"元工具"。

### 5.1 `create_plan`（Planning 节点使用）
- **调用方**：Planning 节点的 LLM
- **入参**：`steps: [{id, tool_name, tool_args, reason, depends_on}]`
- **执行**：将 steps 写入 `state.plan`，设置 `state.current_step = 0`
- **返回**：`"计划已创建，共 N 步"`
- **失败处理**：JSON 解析失败或格式不正确 → 重试 Planning 最多 3 次

### 5.2 `evaluate_step_result`（Execution 节点使用）
- **调用方**：Execution 节点每步执行完后的 LLM 审视
- **入参**：`decision: "continue" | "replan" | "skip"`，`reason: str`，`replanned_steps: [...]`（仅 decision=replan 时）
- **执行**：更新 state 并触发条件路由
- **说明**：`replan` 时仅保留已完成的 Step，未执行的丢弃并替换为新 plan

### 5.3 Summarization 节点
- **不走工具**：LLM 直接输出纯文本作为 `state.final_answer`
- **职责**：综合所有工具结果，给出完整回答，引用来源，不确定时坦诚表述

## 六、系统提示词体系

### 6.1 分层结构

```
system_prompt.yaml
├── base:           角色定义、通用行为规则
├── planning:       extends base + 规划规则（如何拆解问题、选择工具、排顺序）+ create_plan 工具
├── execution:      extends base + 执行规则（如何审视结果、决策 continue/replan/skip）+ evaluate_step_result 工具
└── summarization:  extends base + 回答规则（综合来源、引用出处、不确定时坦诚）
```

### 6.2 工具描述注入

- 工具定义与 Prompt 分离
- 工具 XML 描述从代码自动生成（函数签名 + docstring → Anthropic XML）
- Planning 和 Execution 的 Prompt 中动态注入当前可用工具的描述

## 七、SSE 流式事件

端点 `POST /chat/agent/query/stream` 返回以下事件类型：

| 事件类型 | 含义 | payload |
|---------|------|---------|
| `plan_created` | 计划制定完成 | `{steps: [...], total_steps: N}` |
| `step_start` | 开始执行某步 | `{step_id: str, tool_name: str, reason: str}` |
| `step_done` | 某步执行完成 | `{step_id: str, status: "done"\|"failed"\|"skipped"}` |
| `step_replan` | 计划被修正 | `{old_steps: [...], new_steps: [...], reason: str}` |
| `answer_start` | 开始生成最终回答 | `{}` |
| `delta` | 回答文本增量 | `{content: str}` |
| `done` | 流结束 | `{}` |

前端可基于 `plan_created` 展示检索进度侧边栏。

## 八、容错与安全

| 场景 | 处理方式 |
|------|---------|
| 工具执行报错 | 错误信息作为 tool result 返回给 LLM 审视，由其决策是否重试/跳过 |
| Planning 返回格式错误 | 重试 Planning 最多 3 次 |
| replan 超过 3 次上限 | 强制进入 Summarization，带着已有结果回答 |
| LLM 不调任何工具（直接回答） | 允许——若问题不需要检索则直接回答 |
| 单次 LLM 调用超时 | 重试最多 2 次 |
| 全局 Agent loop 超过 120s | 中断，返回已收集结果 |
| SQL 注入 | SQL Query 工具内部：自然语言→SQL + 仅允许 SELECT + 强制注入 user_id |
| 路径遍历 | OCR 工具层校验文件路径合法性 |
| 跨租户数据泄露 | user_id 由系统从 ContextVar 注入，LLM 不可传 |

## 九、会话持久化

- **消息历史**：完整的 Anthropic content blocks 序列（含 `tool_use` / `tool_result`）JSON 序列化存入 `ChatMessage.content`
- **思考事件**：SSE 事件（`plan_created`、`step_start`、`step_done` 等）持久化到 `ChatThinkingEvent` 表，支持前端刷新后重放
- **现有表结构**：复用 `chat_sessions`、`chat_messages`、`chat_thinking_events`，不新增表

## 十、测试规划

### 10.1 测试原则

- **严格 TDD**：先写测试，再写实现
- **不使用 mock**：LLM 真实调用，工具连接真实 Milvus/MySQL/PaddleOCR
- **自行验证**：Agent 执行完成后自行查询数据库验证数据正确性

### 10.2 测试分层

| 层 | 目录 | 内容 | 数量 |
|---|---|---|---|
| 工具单元测试 | `tests/unit/` | 每个工具函数：插入测试数据 → 调用 → 断言返回 | 8 个文件 |
| Agent 集成测试 | `tests/integration/` | 完整 Plan-then-Execute 流程，覆盖 30 个测试问题 | 9 个文件（按问题分组） |
| 冒烟测试 | `tests/smoke/` | HTTP 端点 SSE 流格式验证 | 1 个文件 |

### 10.3 测试数据

- 测试文件来源：`docs/简历合集/`（11 个文件：1 md + 2 pdf + 1 pptx + 7 docx）
- 数据准备：
  - **工具单元测试**：直接调内部方法（`MilvusService.insert_documents()` 等）上传 + setup/teardown 清理
  - **Agent 集成测试**：通过 HTTP API（`POST /knowledge/upload`）上传，模拟真实用户行为
  - **分组共享**：每个集成测试文件共享一份数据（`scope="module"`），同文件内的多个测试问题复用，减少重复上传耗时
- 30 个测试问题定义：`docs/简历合集/测试问题.md`

### 10.4 集成测试分组

| 文件 | 覆盖问题 | 测试类型 |

### 10.5 断言策略

- **结构化规则断言**：每个测试用例定义一组 `check` 规则
  - `must_contain: [str, ...]` —— 回答必须包含的关键词/实体
  - `must_not_contain: [str, ...]` —— 回答不得包含的内容
  - `must_match: regex` —— 精确正则匹配
- 规则从 `测试问题.md` 的"预期"字段翻译而来
- 并发限制：`pytest -n 2`（避免 LLM API 并发过高触发限流）

## 十二、约束与边界

- **不修改 RAG 流水线**：`RagService` 保持 OpenAI 协议，不做结构性改动
- **不修改前端**：仅 SSE 事件类型变化（向后兼容扩展），不改变前端路由
- **不修改基础设施**：MySQL、Redis、Milvus 配置不变
- **不修改 DjangoUserService**：用户认证体系不动
- **可破坏性重构**：`app/agent/` 目录内全部重写，旧 `agent.py`、`agent_tools.py` 直接替换
