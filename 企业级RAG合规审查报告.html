关键项详证

3.1 长问题处理 — 压缩、改写、HyDE 三项齐全 已实现A
证据：query_processor.py
• 压缩：COMPRESS_PROMPT（L11）+ _compress()（L64），超过 max_length=400 触发
• 改写：DECOMPOSE_PROMPT（L15）+ _decompose()（L74），拆成最多 max_queries 个子问题
• HyDE：rag_service.py _generate_hyde()（L281-292），生成假设性回答作为检索变体，L114 与查询变体合并
三者均有独立超时（预处理 20s / HyDE 15s），超时降级为原始查询。
3.4 多因素加权排序 — 与文档公式精确吻合 已实现A
证据：multi_factor_ranker.py
• 权重：L21-23 w_relevance=0.5, w_time=0.3, w_weight=0.2（与文档公式一致）
• 时间衰减：L73-75 time_decay = math.exp(-lambda * years_ago) 指数衰减
• 文档权重：L78-82 类别权重×0.7 + 质量评分×0.3
• 反聩联动：L44-64 从 DB 查 DocWeight 表读取学习到的权重

评价：文档给出的公式"最终得分 = 相似度×0.5 + 时间衰减×0.3 + 权重×0.2"在本项目中精确落地，且额外引入了贝叶斯 CTR 学习的动态权重。
3.5 混合检索 — Milvus 原生 hybrid + 动态权重 已实现A
证据：milvus_hybrid_retriever.py 同时构造 dense 向量检索（L50-56，COSINE）和 sparse BM25 检索（L59-65），L75 调用 client.hybrid_search() 融合。BM25 通过 Milvus 2.5 原生 Function 实现（milvus_store.py L92-99），中文分词用内置 chinese analyzer。支持 RRFRanker（L68）和 WeightedRanker（L70）切换，并有 get_dynamic_weights() 按查询特征（编号/日期/邮箱）自动调整权重。
3.7 负反馈学习 — 完整闭环 已实现A
证据：完整闭环链路
① 采集：feedback_router.py 三个端点收集 like/dislike/skip + 停留时长 + 点击文档
② 学习：feedback_service.py L69-89 贝叶斯平滑 CTR（prior 1/2）+ 即时反馈调整，新权重 = CTR×0.6 + 历史权重×0.2 + 即时反馈×0.2，钳制 [0.1,1.0]
③ 回流：multi_factor_ranker.py L49-62 从 DB 读取 DocWeight 应用到综合得分
④ 日志：rag_service.py L224-226 异步写 QueryLog（不阻塞主流程）
评价：文档要求的"从用户行为中学习 → 优化排序"完整落地，贝叶斯平滑避免冷启动偏差，是超出多数 RAG 项目的自优化能力。
3.6 查询扩展 — 缺静态同义词词典 部分实现B+
证据：query_processor.py L20-22 EXPAND_PROMPT + _expand()（L87-97）用 LLM 生成语义等价变体，L46-50 对所有子问题并行扩展。但全局无静态同义词词典/领域术语映射表。

五、关键差距与改进建议优先级

以下按优先级排列，P0 为达到企业级 A 线最需补齐的项：

P0
集成 PDFHeadingDetector 到处理管线（数据层面 2.1.8）
现状：代码完整有测试但是死代码，PDF 章节结构信息丢失。
建议：在 pdf_multimodal_loader.py 文本提取阶段调用 build_heading_path()，为每个文本块附加标题路径上下文。工作量小，收益高。
P0
接入 Prometheus 监控指标（工程化 6）
现状：无 /metrics 端点，无请求计数/延迟/错误率指标，无法做 SLI/SLO。
建议：接入 prometheus-fastapi-instrumentator，暴露 /metrics，补充 RAG 检索耗时/召回数/Rerank 耗时等业务指标。
P1
启用语义切分或明确移除（数据层面 2.1.10）
现状：_optimize_chunks() 实现了基于 embedding 的语义合并但未调用。
建议：在 split_documents() 中启用，或移除避免误导。前者提升切分质量，后者减少维护负担。
P1
补充 Embedding 输入长度控制（数据层面 2.2.1）
现状：bge-large-zh 最大 512 token，无 token 级截断。
建议：在 encode 前用 tokenizer 截断至 480 token 留余量，查询时同理。
P1
落地缓存装饰器或移除死代码（工程化 3）
现状：cache_with_redis 装饰器未使用，RAG 热点未缓存。
建议：应用于 Embedding 结果/高频查询检索结果缓存；清理 print() 调试语句。
P1
修正限流实现（工程化 2）
现状：注释称"令牌桶"实为"固定窗口"；全局限流中间件被注释。
建议：改用滑动窗口或令牌桶；启用全局限流；修正注释。
P2
完善应用容器化（工程化 9）
建议：加 gunicorn -k uvicorn.workers.UvicornWorker --workers N 多进程；加非 root 用户；加 HEALTHCHECK；docker-compose 补 backend 服务实现一键部署；保留 uv.lock 保证可复现。
P2
补齐工程化模块测试（工程化 8）
现状：rate_limit/auth/agent/session_manager/exception_handlers 零覆盖。
建议：优先补 auth_utils（鉴权是安全核心）和 exception_handlers 测试。
P2
配置安全收紧（工程化 7）
现状：db_config.py echo=True 生产风险；main.py CORS allow_origins=["*"]+allow_credentials=True 不安全组合；chat.py L76 /chat/sessions 缺鉴权。
建议：生产关 echo；CORS 限定白名单；补 /chat/sessions 鉴权依赖。
P2
扩充 OCR 纠错词表 / 补充领域同义词词典（数据层面 2.1.6 / 检索 3.6）
建议：纠错字典从 11 条扩充至百条级或集成纠错模型；补充企业垂直领域同义词表作为查询扩展兜底。

六、总结论

审查结论

对照《企业级 RAG》文档定义的标准，本项目后端在数据层面与检索策略两大核心维度已全面达成企业级要求，且多项实现超出文档预期：

  • 文档要求的 Excel 合并单元格/多级表头、代码 AST 切分、PDF 多模态处理、三种图片向量化方法全部实现，部分有完整测试
  • 检索策略的 7 个核查点中 6 项完整实现，多因素排序公式与文档精确吻合，负反馈学习形成完整闭环
